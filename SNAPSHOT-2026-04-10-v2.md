## What This Session Produced

A concept-level redesign of sevens' architecture, captured in five
documents under `docs/design/`. The redesign was derived from the
existing responsibility catalog and intent document through discussion,
not from reading code. The goal is to establish clean concept boundaries
that resolve the structural problems documented in the prior snapshot
and code review.

## Concept Architecture

Four concepts and one architectural contract, organized in layers:

### Layer 1: Graph (`concept-graph.md`)

Triple store. Assert, retract, query. No semantics beyond "triples
exist." This is the computational realization of Category K's single
axiom: knowledge is synchronic binary relation.

State: a set of triples, each with (subject, predicate, object).
Unique on the full 3-tuple. Idempotent assertions, thinness (no
metadata on relations).

Design invariants: uniformity (everything is a triple), open predicate
space (new information is new predicates, no schema migration),
thinness (relations hold or don't, no weights or annotations).

### Layer 2: GraphOps (`concept-graph-ops.md`)

Predicate metadata, path composition, and structural operations.
Sevens-agnostic -- reusable outside the PKM domain (e.g., in a
Haskell exploration of these ideas).

State: predicate specifications (functional vs. relational,
inverses, symmetry, transitivity). This is the metadata layer 1
doesn't carry.

Key operations: `set` (functional predicate write), `compose`
(multi-hop path walking -- arrow composition), `reachable`
(transitive closure), `subgraph` (subtree extraction).

Separated from layer 1 so the categorical machinery (how morphisms
compose, what properties predicates have) is decoupled from bare
storage.

### Layer 3: KnowledgeBase (`concept-knowledge-base.md`)

The PKM domain model. What makes a graph of triples into a sevens
knowledge base.

Establishes:
- Subject identity scheme (`node:<hash>:<title>`)
- Core predicate vocabulary (`node/parent`, `node/content`,
  `node/link`, `block/node`, `block/content`, etc.)
- Session/focus predicates (`session/focus`, `session/include`, etc.)
- Log predicates (`log/function`, `log/node`, `log/timestamp`, etc.)
- Structural invariants (tree property, acyclicity, 7+/-2 constraint)
- Queries (walk, overview, children, siblings, validate)

Key design decisions:
- Blocks are nodes. Same entity, different predicate namespace
  prefix. The namespace is a label on a composition of morphisms --
  it tells the tool which compositions to follow for which operations.
- Identity is defined by morphisms, not type fields. A node IS the
  pattern of predicates connecting its subject to other subjects.
  The subject string is a handle, not the identity.
- Session, focus, and logging are predicates in this concept, not
  separate concepts. They're the meta-temporal layer: what you did
  with your ideas and in what context.
- Log entries are presentational, not operational. They record what
  happened but don't drive behavior. The Function concept owns its
  own operational state.

### Function (`concept-function.md`)

Typed transformations on the knowledge base, composed as curried
pipelines with configurable control flow and review gates.

A function is not "an AI operation." It is a typed arrow: input
context (a subgraph gathered by composing predicates) goes in,
structured output (graph mutations, text, or both) comes out. The
transformation is a black box -- could be an LLM, a deterministic
computation, or an external agent.

Key structures:
- **Signature**: input/output contract, backend-agnostic. The
  function declares what it consumes and what shape the result must
  have. This is what makes composition type-checkable.
- **Backend spec**: how a particular executor fulfills the contract.
  LLM backends get prompt templates. Deterministic backends get
  handler references. Agent-mode backends get the signature itself.
- **Pipeline**: curried function composition. Each step is a partial
  application point. The pipeline remembers where it paused and
  can resume across process restarts.
- **Gate**: configurable review state machine at each curry point.
  Pending/accept/reject/revise. Configurable per step: revisable,
  history policy (none/latest/full), cancelable, auto-accept,
  rollback-on-reject.
- **Control flow**: sequence, loop (user-terminated or
  condition-terminated, with replace or append accumulator), branch
  (conditional on step output), delegation (composedOf), and
  map-over (functor).

Pipeline state is operational, owned by Function. Log entries are
written as side effects of pipeline transitions via sync to
KnowledgeBase.

### Projection (`concept-projection.md`)

Architectural contract (not a user-facing concept -- transparent to
the user). The natural transformation between graph semantics and
presentational surfaces.

Parameterized by format: markdown, org-mode, webapp, etc. Each
format implementation satisfies the same contract: render (graph ->
presentation), parse (presentation -> triples), reconcile (diff and
merge), sync (full round-trip), write (graph -> presentation push).

Data authority model:
- DB is authoritative for persistence (saved state)
- File on disk is authoritative for current content on conflict
  (working state, like an unsaved form)
- Git is checkpoint history, tied to the markdown projection
  specifically
- On irreconcilable conflict, file wins; DB reconstructs to the
  last reconcilable point, not the full history
- Full re-sync from files is the fallback of last resort

Git is part of the markdown projection, not a separate concept.
A webapp projection wouldn't use git.

## Concepts Absorbed

The following initially looked like separate concepts but collapsed
into configurations of the four above:

| Originally | Absorbed Into | How |
|---|---|---|
| Suspension | Function | Pipeline gate state (Pending/Accepted/Rejected) |
| Discussion | Function | Looping gated step with append accumulator |
| Templates | Function | Deterministic backend fulfilling a function signature |
| Session/Focus | KnowledgeBase | Predicates (`session/focus`, `session/include`, etc.) |
| Logging | KnowledgeBase | Predicates (`log/function`, `log/timestamp`, etc.) |
| Git | Projection | Format-specific (markdown projection owns git) |
| Agent mode | Function | TransformBackend choice |

Started with ~10 candidate concepts, ended with 4 + 1 contract.

## How This Resolves Known Issues

The prior snapshot and code review documented trust-sensitive workflow
problems. The concept architecture addresses each:

**Pending ambiguity**: pipeline state is owned by Function with an
explicit state machine. Each pipeline has an ID, a current step, and
a state. No ambiguity about which pending item is being acted on.

**Discussion cancel semantics**: discussion is a looping function
with a cancelable gate. Cancel means rollback per the GateSpec
configuration. One behavior, defined once.

**Backend continuity across multi-step flows**: TransformBackend is
a stateless-per-call interface. The pipeline owns all state; the
backend is called fresh each step. No backend state to lose.

**Revision lifecycle divergence (CLI vs. REPL)**: revision is a
Function action (`revise`) operating on pipeline state. One
implementation. The CLI and REPL are presentation layers that call
the same action.

**Multi-root identity safety**: KnowledgeBase scopes subjects by
root hash. The identity scheme is defined once in the concept, not
reimplemented per caller.

**`internal/apply` having 14+ responsibilities**: the concept
boundaries suggest a clean package mapping where each package has
one purpose. The current `internal/apply` responsibilities distribute
across Graph, GraphOps, KnowledgeBase, Function, and Projection.

## Layer Model

```
Layer 4: User-defined types and semantics
         (custom predicates, DSL types, domain structure)
         Not a concept -- the open predicate space that users fill
------------------------------------------------------------
         Function
         (typed transformations, pipelines, gates, control flow)
------------------------------------------------------------
Layer 3: KnowledgeBase
         (PKM domain model, predicates, identity, session, logs)
------------------------------------------------------------
Layer 2: GraphOps
         (predicate metadata, path composition, structural ops)
         Sevens-agnostic, reusable
------------------------------------------------------------
Layer 1: Graph
         (triple store: assert, retract, query)
         Category K floor
------------------------------------------------------------
         Projection (contract)
         (natural transformation: graph <-> presentational surface)
         Parameterized by format. Transparent to user.
```

Projection sits alongside the stack rather than in it -- it bridges
graph state to any presentational surface without belonging to a
specific layer.

## Open Questions Remaining

Across all concept documents, the following questions are unresolved:

1. **Transaction boundaries** (Graph): should the triple store have
   atomic multi-triple transactions?
2. **Search placement** (Graph/GraphOps): substring search is
   predicate-unaware but could belong to layer 2.
3. **Bootstrap problem** (GraphOps): if predicate specs are stored as
   triples, the meta-predicates need their own specs.
4. **Delete semantics** (KnowledgeBase): cascade, orphan, refuse, or
   reparent on node deletion?
5. **Rename semantics** (KnowledgeBase): renaming changes subject
   identity. Should identity be decoupled from title?
6. **`node/file` ownership** (KnowledgeBase/Projection): is the file
   path a KB predicate or a projection predicate?
7. **Context policy** (Function): keep the minimal/neighborhood/full
   shorthand or eliminate in favor of explicit path specs?
8. **Agent mode async** (Function): does the async nature of
   prepare/submit require a distinct pipeline state ("waiting for
   external submission")?
9. **Signature representation** (Function): JSON schema, or subgraph
   shapes per Category K?
10. **Multiple simultaneous projections**: conflict resolution when
    two projections edit the same node.

## What Comes Next

This concept architecture is a design artifact, not code. The next
steps are:

1. Review the concept documents for coherence and completeness
2. Decide on open questions (especially delete semantics, rename/
   identity, and transaction boundaries)
3. Map current code to concept boundaries to identify what moves where
4. Plan the refactoring sequence -- which concept boundary to
   establish first, given the current code structure
5. Begin implementation, starting from the bottom (Graph, GraphOps)
   and building up

The concept design was done without reading code intentionally. The
refactoring plan should be done WITH the code, mapping the current
responsibilities catalog to the new concept boundaries.
