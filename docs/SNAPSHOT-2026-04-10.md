## Executive Summary

Sevens is no longer just a minimal graph CLI with a few AI functions. The current codebase and binary surface show a much broader local-first thinking tool:

- a Markdown knowledge graph backed by a triples store
- a node-level and block-level graph/query layer
- an AI function pipeline system with suspensions, approvals, revisions, and agent-mode round trips
- a deterministic template/manual capture layer
- an interactive REPL with focus state and specialized modes
- multiple backend paths for inference and agent integration

The product is meaningfully ahead of the oldest specs and also somewhat ahead of the `README.md` in certain manual/block workflows.

At the same time, the repository's own review and exploratory test reports describe unresolved workflow-integrity risks:

- pending item ambiguity
- discussion cancel semantics
- backend continuity across multi-step flows
- git staging/revert semantics
- multi-root identity safety

The net result is:

- conceptually: ambitious and fairly coherent
- structurally: substantial and real, not just a prototype sketch
- operationally: promising, but still carrying trust-sensitive edge cases
- documentation-wise: rich but layered, with a mix of current, stale, and aspirational material that needed consolidation

## Current Functional Shape Of The Tool

Based on the regenerated API map, current binary help, and existing architecture docs, the present tool appears to consist of the following major subsystems.

### 1. Triple-store graph core

The system has converged around a triple-store model:

- node and block facts live as `(subject, predicate, object)` strings
- graph traversals are built from triple lookups and composition helpers
- search, lineage, suspensions, and other metadata also flow through the store

This is no longer just a conceptual aspiration in the design docs. It is the operating architecture described by the current generated API and architecture notes.

### 2. Markdown sync and graph derivation

The graph layer reads Markdown files, parses frontmatter and wiki-links, and populates derived graph facts.

The presence of both node-level and block-level commands strongly suggests the tool has moved beyond simple file-as-node inspection and now treats internal structure as a first-class surface in several workflows.

### 3. AI function system

The function system currently supports:

- single-step and multi-step functions
- path-based and role-based context resolution
- suspensions/gates
- revisions
- composed functions
- map-over execution
- agent-mode `prepare` and `submit`

This is a fairly sophisticated orchestration layer, not just prompt templating.

### 4. Backend abstraction

The current generated API and architecture docs indicate three backend paths:

- Anthropic
- Codex
- Claude

That puts sevens in a hybrid position:

- it can operate as a direct LLM client
- it can also operate as a task compiler/runtime around external coding-agent CLIs

### 5. Deterministic template/manual layer

The command surface now clearly exposes:

- `templates`
- `capture`
- `instantiate`
- `new`

This means the manual/deterministic construction path is no longer just a plan. It is current product surface.

### 6. REPL

The REPL is present as a real entry point with focus support and its own help surface.

The existing design docs suggest a fairly rich readline-based REPL with:

- navigation
- note mode
- discussion mode
- template helpers
- review/accept loops

The generated API map also shows dedicated REPL handlers for capture, template invocation, note/discussion flows, and block focus behavior.

## Leading Design Direction

The leading architectural direction for sevens appears to be this:

- the database becomes the most important candidate for the authoritative source of truth
- the product is understood increasingly as the database schema plus the operations over it
- Markdown and git are treated increasingly as projection and translation layers, not as the core product

That is not just a storage preference. It is a strong reframing of what sevens might become.

In that framing:

- node content would live authoritatively in the DB
- block structure and semantic relationships would live authoritatively in the DB
- morphism history, suspensions, lineage, and operational state would live authoritatively in the DB
- the CLI would be understood as an interface onto DB-backed operations
- Markdown would be a user-editable view
- git would be a versioned transport and audit view over one projection of the data

The key requirement in this direction is bidirectionality.

Sevens would not merely export Markdown from the DB. It would also need to be able to:

- observe file edits
- observe git-level changes
- interpret those as edits to the projected view
- reconcile them back into the same base logic and schema

That means the file and git layer would be treated as one translation surface among others:

- DB -> Markdown
- Markdown -> DB
- DB -> git-tracked filesystem state
- git/file changes -> DB updates interpreted through the same graph and workflow rules

If the project continues leaning that way, the central question is no longer "why is the DB larger than the Markdown?"

The real questions become:

- is the DB carrying the right primary information?
- are the operations defined against the DB first, rather than against one projection?
- do projections like Markdown preserve enough structure to round-trip cleanly?
- is morphism history modeled as part of the product rather than as disposable local cache?

This matters because the current implementation still shows a hybrid posture:

- files are still treated as source-of-truth for note content in many places
- the DB is already source-of-truth for logs, suspensions, and runtime state
- git is part provenance layer, part transport layer, part recovery mechanism

The cleaner version of that direction looks like:

- one primary model
- many projections
- Markdown is the most important projection, because it is human-editable and durable
- git is the most important transport/audit projection, because it moves and versions that human-editable view

If sevens keeps moving in that direction, DB heft is not itself a design smell. The DB starts looking much more like the product.

At that point, storage questions are better evaluated against the needs of the authoritative model:

- content
- structure
- block identity
- provenance
- morphism history
- pending workflow state
- reversible operations

The smell would not be "the DB is bigger than the Markdown." The smell would be:

- duplicated state that does not serve the authoritative model
- projections that cannot round-trip cleanly
- machine-local history that ought to be portable per root
- logic defined against files first rather than against the DB model first

This is the right design lens for future work on:

- portable morphism history
- migration to new machines
- multi-root portability
- remote/ref metadata per root
- sync and transport concerns
- whether `sevens.db` is cache, source-of-truth, or a mixture of both

## Current Product Position Relative To Earlier Snapshots

Compared to the earlier snapshots now under `docs/`, the tool has clearly crossed several thresholds:

### It is no longer node-only

The binary now exposes block-aware flows directly:

- inspect block structure
- diff block structure
- extract a block into its own node

That is a meaningful evolution from the earlier node-centric snapshots.

### It is no longer AI-only in spirit

The manual structure/capture/template path is now a real peer to the AI function path:

- `capture`
- `templates`
- `instantiate`
- `new`

This is important productually. Sevens is now a mixed-mode tool:

- deterministic where structure should be deterministic
- AI-mediated where judgment/synthesis helps

### It is no longer just "call a model"

The function/runtime system has become a workflow engine:

- approvals
- rejections
- revisions
- pending state
- agent-mode checklists
- composed functions
- map-over semantics

### It still carries prototype-era trust issues

The strongest risks are no longer about "does the idea work?" They are about:

- whether the workflow does exactly what the UI says
- whether the right pending item is being accepted
- whether cancel and revert mean what users think they mean
- whether repo history is changed more broadly than intended

That is a healthier class of problem than "nothing exists", but it is still a serious class of problem.
