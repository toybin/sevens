# Sevens — Project Snapshot (v2)

**Date**: 2026-04-07 (evening)
**Lines of Go**: ~3,300 across 14 files
**Status**: Working prototype with expanded function library, tested end-to-end on a real knowledge graph

## What Changed Since Last Snapshot

### New Capabilities

- **12 functions** (up from 3): decompose, elaborate, summarize, discuss, notice, challenge, distill, bridge, relate, synthesize, merge, thesis
- **Focus sessions**: `sevens focus <node>` pins a node with `--include`/`--exclude` for persistent context. `.` shorthand for focused node in all commands.
- **Pre-call cost gate**: Token estimation + cost confirmation before every LLM call. Auto-approves below configurable threshold (default $0.01). `--yes`/`-y` to skip.
- **Streaming output**: Text-type functions stream to terminal in real-time. Ops/suggestions suppress streaming to avoid partial JSON.
- **Direct API key**: `:api-key` in config.edn alongside `:api-key-env`. Named model profiles (`:models {:fast {...} :capable {...}}`).
- **Discussion continuation**: `discuss` detects existing discussion child and appends new agent turns responding to user entries instead of creating a new node.
- **`--include` on apply**: One-shot context injection without changing focus session.
- **`sevens tree <node>`**: Subtree view rooted at any node.
- **`sevens log <node>`**: Operation history with compact display.
- **`sevens status`**: Shows focus state, includes, context files.
- **Pending summaries**: `sevens pending` shows function name, step, and content summary.
- **Validation hints**: Overflow and length warnings now suggest `decompose` as remediation.

### Bug Fixes

- Non-ops output types (text, suggestions) no longer parsed as JSON ops on last step
- Non-ops last steps log as "completed" instead of "suggested" — don't create false pending state
- YAML titles with colons are quoted in generated frontmatter
- LLM-generated content has frontmatter stripped before `createFile` wraps it (prevents double frontmatter)
- Failed ops execution no longer consumes pending state — "accepted" logged only after successful ops
- Root inferred from active focus session when no `.sevens.edn` found
- Streamed text output no longer double-printed (stderr stream + stdout echo)

### Refactoring

- `samber/lo` adopted for `ListFunctions` (FilterMap) and `RenderStepPrompt` (Reduce over replacements)
- API key resolution extracted to `resolveAPIKey` helper
- `ResolveModel` method on GlobalConfig for tier-based model lookup

**Workflow tested**: braindump → notice → discuss → user responds → discuss continues → decompose (with revision) → distill discussion into siblings → challenge → bridge → synthesize → thesis → decompose overflow node → relate → tree view

**Context files tested**: Global author bio injected into all calls. Per-node Portland zoning notes injected into space design calls. Both verified working — LLM referenced bio content unprompted.

## Gaps Identified From Testing

### 1. The Context Problem (Critical)

Every function that needs to read or edit nodes other than its target fails or produces garbage without manual `--include`. This was the #1 source of failures during testing.

**What happens**: `distill` reads a discussion node and tries to edit the parent and siblings. But it can only see the target node's content — it has to guess what the parent/siblings say. It hallucinates `old_text` that doesn't exist, and the edit fails. Same for `bridge` (needs sibling content), `relate` (needs graph-wide awareness), and `merge` (needs children content).

**The workaround**: `sevens focus "X" --include "Y" --include "Z"` or `sevens apply fn "X" --include "Y"`. This works but puts the LLM's job on the user — the user has to figure out what context the function needs.

**The real fix**: The brainstorm's arity-promotion / context-gathering model (§7). The function definition declares what it's allowed to request, a cheap model does a few turns of graph navigation to decide what's relevant, and the gathered context is frozen before the main call fires. Until this is built, every multi-node function is fragile.

**Intermediate fix**: Functions that edit siblings (distill, bridge, relate) should auto-include the parent and all siblings by default. This is a simpler version of context gathering — hardcoded rather than dynamic, but covers the most common case.

### 2. No Undo / Revert

The tool automates the forward path (suggest → accept → apply → commit) but leaves recovery entirely to the user. If an accept produces bad content that isn't caught immediately, the user's options are:

- `git log` to find the pre-accept commit
- `git diff HEAD~1` to see what changed
- `git checkout <hash> -- <file>` to restore specific files
- `sevens sync` to rebuild the database

This works because git is there, but it's manual and error-prone. A `sevens revert <node>` that undoes the last applied operation (using the log's commit hash) would close the loop. The log already records commit hashes — it's a matter of wiring up `git revert` or `git checkout` and re-syncing.

A softer version: `sevens diff <node>` that shows what changed since the last sync/accept, so the user can review before it's too late.

### 3. Fragile Edit Operations

Exact string matching for `old_text` is the second most common failure mode after missing context. The LLM paraphrases, truncates, adds/removes whitespace, or interpolates from memory instead of quoting verbatim. When it fails, the entire accept is blocked.

**Options**:
- **Fuzzy matching**: If exact match fails, try a similarity-based match (Levenshtein, longest common subsequence). Show the closest match and ask the user to confirm.
- **Line-range edits**: Instead of `old_text`/`new_text`, use `line_start`/`line_end`/`new_text`. Less fragile but requires the LLM to count lines accurately (also error-prone).
- **AST-level operations**: Use goldmark to parse the markdown AST and operate on structural elements (insert after heading X, replace list item N). More robust but much more complex.
- **Hybrid**: Try exact match first. On failure, try fuzzy match. On fuzzy match, show the diff and ask for confirmation. This is probably the right first step.

### 4. Functions Can't See Each Other's Output

Running `notice` then `challenge` on the same node, `challenge` has no awareness of what `notice` found. Each function call is stateless — it assembles context from scratch (node content + graph context + context files) with no memory of prior function applications.

The DESIGN.md specifies a `{{history}}` template variable that would inject the node's function application log into the prompt. This would let `challenge` see that `notice` flagged certain assumptions, and target those specifically.

**What `{{history}}` needs**:
- Read the node's log entries
- Filter to recent relevant entries (last N, or since last major edit)
- Format as a readable summary: "notice found: [patterns, gaps, tensions]. synthesize found: [cross-cutting themes]."
- Inject into the prompt where `{{history}}` appears

This is straightforward to implement — the log already has `RawOutput` and `Summary` fields. The question is how much history to include (context window cost) and how to format it (raw output vs. summary vs. structured).

### 5. No Map-Over / Batch Operations

Every function application is one node at a time. To bridge all 9 children of "The Commons," you run the command 9 times. To elaborate all the sparse children of a freshly decomposed node, you run it N times.

The brainstorm's higher-order composition model (§7) handles this: `{:map-over "target.children" :fn "elaborate"}`. But even without the full composition system, a simpler `--each-child` flag would help:

```
sevens apply elaborate "The Commons" --each-child
```

This runs the function on each child of the target, with appropriate gating (confirm once for the batch, not per-child, or set a batch cost threshold).

### 6. Pending State Awkward for Non-Actionable Output

`synthesize` produces suggestions (insights/connections) that sit in `pending` forever because there's no operation to "accept." The user's options are:
- `sevens accept` which does nothing meaningful (logs acceptance, no ops)
- `sevens reject` which discards the insights
- Ignore it, leaving stale pending state

Suggestions output should either:
- Auto-complete like text (log as "completed," display immediately)
- Have a "promote" action that turns a specific suggestion into a new node: `sevens promote "The Commons" --suggestion 3` creates a node from the third suggestion

The promote path is more interesting — it connects the synthesize/notice/thesis functions to the graph-building workflow. Right now insights are write-only; they can't feed back into the structure without manual node creation.

### 7. Discussion Nodes Need Automatic Parent Context

The `discuss` function creates/continues a conversation about a node, but the discussion child's content is just the conversation thread. When continuing a discussion, the LLM sees the discussion node (questions and answers) but not the parent node (the actual content being discussed).

This means the agent's responses drift — it responds to the user's words without grounding in the source material. The fix: `discuss` should always auto-include the parent's content, either by injecting it as a context file or by having the discuss prompt template reference the parent explicitly.

More generally, any function that operates on a "typed" child (discussion, summary, evidence) should auto-include the parent. The child exists in relation to the parent — it doesn't make sense to process it in isolation.

### 8. No Search

With 17 nodes, you can hold the graph in your head. At 50+, you need `sevens search "governance"` to find nodes by content or title. The database has full content in the `nodes` table — a SQL `LIKE` query would work for v1.

```
sevens search "cooperative" → 
  Commons Governance Models (body match)
  Discussion: The Commons (body match)
  Existing Community Infrastructure (body match)
```

Full-text search is a later concern (Turso has native vector search for when embeddings make sense). But basic substring search is table stakes for usability.

### 9. The Tree Is Flat

After decompose → elaborate → decompose, the graph is still mostly two levels deep. The tool doesn't encourage or facilitate depth. There's no "deepen this subtree" operation that recursively scaffolds.

What this might look like: a `deepen` function (or a composed pipeline) that looks at a node with children and:
1. Identifies which children are sparse or stub-like
2. Runs `elaborate` on the sparse ones
3. Identifies which children are over-length or cover multiple topics
4. Runs `decompose` on those
5. Repeats until the subtree reaches a target depth or all nodes are within bounds

This is a natural application of the map-over + composition model from the brainstorm — it's `decompose` and `elaborate` composed with a filter condition.

### 10. No Multi-Node Input

Several desired functions require two or more explicit node inputs:
- **compare**: "(node A, node B) → text" — what do they share, where do they diverge?
- **merge**: "(child[0..N]) → ops" — combine children into parent
- **transfer**: "(source, target) → ops" — move relevant content from one node to another

The current function system is strictly arity-1: one target node, one function. The brainstorm's curried node-typed signatures (§2) address this, but even a simpler approach would help — allow functions to declare additional node arguments:

```
sevens apply compare "Node A" "Node B"
```

The function definition would declare `{:inputs [{:role "left"} {:role "right"}]}` and the prompt template would have `{{left-content}}` and `{{right-content}}`.

### 11. System Prompt Is Output-Type Agnostic

The global system prompt always says "return a JSON array of operations" regardless of the function's output type. This biases text-output functions toward JSON (notice, challenge, thesis all had to fight the system prompt's formatting instructions).

The fix: the system prompt should be parameterized by output type. When calling a text-output function, strip or replace the JSON formatting instructions. When calling an ops-output function, include them.

This could be:
- Multiple system prompts in config (`:system-prompt-ops`, `:system-prompt-text`)
- A template variable in the system prompt (`{{output-format}}`) that's filled based on the function
- Function-level system prompt overrides in the EDN definition

The simplest: let the function's prompt template be authoritative about output format, and make the system prompt generic (structural rules, graph conventions) without output formatting directives.
