Import discipline is clean. No layer violations. Each layer imports only

## Implementation plan: finishing the migration

### Phase 1: Adopt workflow in CLI (high impact, low risk)

Replace inline sync orchestration in `cmd/sevens/pipeline_cmds.go` with
workflow calls. Three commands to change:

1. **`applyCmd2`**: Replace lines 110-147 (inline ops application) with
   `workflow.ApplyFunction`. The function already exists and is tested.
   Display logic stays in the CLI; orchestration moves to workflow.

2. **`acceptCmd2`**: Replace lines 232-313 (backend resolution + accept +
   ops application) with `workflow.AcceptPipeline`. The backend
   persistence fix (BUG-5) means the pipeline already carries its backend
   name -- workflow reads it.

3. **`rejectCmd2`**: Replace lines 326-401 (pipeline finding + reject)
   with `workflow.FindPendingPipeline` + `workflow.RejectPipeline`.

After this, CLI and workflow share one code path. The REPL's
`PipelineRunner` adapter should also be updated to call workflow
functions instead of the executor directly.

**Verification**: Run the walkthrough manually again. The existing 55
tests in walkthrough_test.go cover the underlying operations; the CLI
tests become redundant with the workflow tests.

### Phase 2: Lift discussion mode to workflow (medium impact)

Extract the three discussion syncs from `internal/repl/discuss.go` into
`workflow.StartDiscussion`, `workflow.ContinueDiscussion`,
`workflow.EndDiscussion`. The REPL becomes a thin UI layer that collects
user input and displays agent output.

This makes discussion testable without the REPL (using the walkthrough
test harness with mock backends) and available to the CLI via
`sevens discuss`.

### Phase 3: Decide on agent mode (design decision)

Two options:

**Option A: Implement AgentBackend.** Create a `TransformBackend` where
`Execute` returns the prompt as a structured checklist (the `prepare`
output). The pipeline enters a "waiting for external submission" state.
`submit` provides a `TransformResult` that the pipeline validates and
transitions on. This makes agent mode a backend choice, not a separate
code path.

**Option B: Keep prepare/submit as CLI-only coordination.** Remove
`BackendAgent` from types.go. Document that agent mode is a CLI
convenience layer that renders prompts and accepts results outside the
pipeline state machine. Simpler, but means agent mode can't use gates,
revision, or multi-step pipelines.

Recommendation: Option A, because agent mode is described in the
walkthrough as supporting multi-step functions with gates, which requires
pipeline integration.

### Phase 4: Cleanup (low risk)

- Register `session/root` predicate in `kb/predicates.go`
- Update WALKTHROUGH.md block path format (`p.0` -> `0`)
- Centralize pipeline transition logging in executor
- Remove stale snapshot references from other docs
- Complete EDN-to-graph session migration (remove EDN file dependency)

### Phase 5: Re-test discuss with claude backend (BUG-10)

After phases 1-2, run `sevens discuss` with `--backend claude` and
examine the subprocess stderr. The fix is likely in how
`ClaudeBackend.Complete` constructs the subprocess command for
multi-turn prompts.
