# Sevens — Project Snapshot

**Date**: 2026-04-07
**Lines of Go**: ~2,600 across 11 files
**Status**: Working prototype, actively evolving

## What's Implemented and Working

### Core Graph Engine
- **Sync**: Scans `.md` files recursively, parses YAML frontmatter (title, parent, max-chars, context-files), extracts `[[wiki links]]` from body, rebuilds SQLite database via Turso driver
- **Overview**: ASCII tree view of the full graph hierarchy with validation warnings (orphans, missing parents, overflow, length violations)
- **Walk**: Shows a node's content plus its neighborhood (parent, children, siblings, cross-refs)
- **Validation**: Orphan detection, missing parent detection, branching overflow (>9 children), character limit violations with cascading `max-chars` inheritance

### Multi-Root Architecture
- Central database at `~/.config/sevens/sevens.db` — multiple roots coexist
- `.sevens.edn` in a directory marks it as a sevens root (with portable `~/` paths, alias, max-chars)
- Root auto-discovery: walks up from cwd looking for `.sevens.edn`
- `sevens sync` with no args syncs all registered roots from `~/.config/sevens/roots.edn`
- `sevens init` creates a new root with `.sevens.edn`, `git init`, and first sync

### Functions System
- **Function definitions**: EDN metadata + separate `.md` prompt template files in `~/.config/sevens/functions/`
- **Typed inputs/outputs**: Steps declare `input` and `output` types (`"node"`, `"suggestions"`, `"ops"`, `"text"`). Composition validated at load time.
- **Multi-step pipelines**: Steps chain sequentially, with `{{prev}}` carrying output forward
- **Gates**: `:approve` pauses the pipeline for human review
- **Suggest → Review → Accept cycle**: `apply` produces suggestions, `accept` advances pipeline or executes ops, `reject` discards
- **Revision with history**: `accept --with "feedback"` re-runs the current step with full conversation thread (all previous suggestions + revisions)
- **Three built-in functions**: `decompose` (two-step: suggest titles → generate content), `elaborate` (single-step: expand sparse content), `summarize` (single-step: condense)

### LLM Integration
- Anthropic API via official Go SDK (`github.com/anthropics/anthropic-sdk-go`)
- Streaming responses (accumulated silently, not dumped raw to terminal)
- System prompt from `~/.config/sevens/config.edn` sent on every call
- `--dry-run` flag to preview rendered prompts without LLM call
- Truncated JSON recovery (closes partial arrays when response hits max_tokens)

### Context Files
- **Global**: `context-files` in `~/.config/sevens/config.edn` — injected into every LLM call
- **Per-function**: `context-files` in function EDN — injected when that function runs
- **Per-node**: `context-files` in node YAML frontmatter — injected when operating on that node
- All three layers merge. Contents wrapped in `<context-file path="...">` XML tags.

### Git Integration
- `sevens sync` auto-commits dirty changes before rebuilding the database
- `sevens accept` commits applied operations with structured message (`sevens: apply <fn> to "<node>"`)
- Log records git commit hashes for provenance
- `sevens init` runs `git init` if not already a repo

### CLI
- Built with Cobra — shell completions for fish/bash/zsh
- Dynamic tab-completion for node titles (queries DB) and function names (scans function dir)
- Human-readable output: ASCII tree for overview, metadata header + content for walk, numbered suggestion lists, ops summary
- `--edn` flag on overview and walk for machine-readable output
- Linked to `~/.local/bin/sevens` for global access

### Logging
- Append-only EDN log per node at `~/.config/sevens/log/<root-hash>/<node>.edn`
- Events: `suggested`, `revision`, `accepted`, `applied`, `rejected`
- Tracks pipeline step, step index, raw LLM output, parsed ops, commit hash, affected files
- `sevens pending` shows nodes with unresolved suggestions

### Obsidian Compatibility
- Parent field uses `[[wiki link]]` syntax in frontmatter: `parent: "[[Parent Title]]"` — stripped during parsing
- Files work as a standard Obsidian vault with wiki links rendering correctly

## What's Designed but Not Built

### From DESIGN.md
- **Richer input context**: `{{history}}` (function application log for the node), `{{graph}}` (broader ancestor/sibling context beyond immediate neighborhood)
- **Higher-order composition**: Map a function over children, filter nodes conditionally, fold/aggregate across nodes
- **Gate types beyond `:approve`**: `:fill` (template with holes), `:select` (pick from options), `:edit` (modify draft in editor)
- **Cross-root function application**: Apply functions to nodes in a different root
- **Effect declarations**: Formal specification of what a function does (creates files, edits files, no side effects)

### Discussed but Unresolved
- **Haskell rewrite**: Considered switching to Haskell for stronger type-theoretic foundations (GADTs for composition checking, monadic effects). Decision: prototype the type theory concepts first as a design exercise, keep Go for the working tool. May revisit for a v2.
- **VCS abstraction**: Currently git-only. jj (Jujutsu) support discussed — would be a thin shim since jj uses git storage. Not started.
- **AST-level section targeting**: Discussed using goldmark's AST for structural markdown operations (insert after heading, replace list item). Current edit model is exact string replacement. AST work deferred.
- **Annotation system**: In-file or sidecar annotation markup that links content spans to the type/morphism system. Conceptual only — no design settled.
- **Markdown extension**: Custom annotation syntax linking content to the function type system. Very future.

## Architecture Decisions Made

1. **Turso driver** (`turso.tech/database/tursogo`) — no CGO, standard `database/sql`, ready for remote sync later
2. **EDN for config/log, YAML for frontmatter, markdown for prompts** — each format where it's strongest
3. **Central DB, not per-project** — `~/.config/sevens/sevens.db` with root-scoped data
4. **Filesystem is NOT the tree** — frontmatter `parent` field defines the hierarchy, files can live anywhere
5. **Functions as typed arrows** — input/output types on steps, composition validated at load
6. **Asynchronous human-in-the-loop** — suggest → review → accept, mediated by files and CLI, not chat
7. **Git for diffs, EDN log for provenance** — don't reinvent version control
8. **Prompt templates as separate `.md` files** — no escaped strings in EDN config
9. **Edit operations as exact string replacement** — same semantics as agent CLI edit tools
10. **Cobra over Kong** — switched for shell completion support
